**Project rule (acapella):** 코드 변경 전 반드시 분석을 먼저 제시하고 사용자 확인 대기.

## 순서
1. **증상/현상** — 무엇이 잘못되었는지 정확히 기술
2. **가능한 원인** — 가설들, 영향 범위
3. **변경 계획** — 수정할 파일/함수, 접근 방식, 대안 비교
4. **예상 리스크** — 부작용, 회귀 가능성
5. **사용자 확인 대기** → 동의 시 코드 수정

## 예외
- "바로 진행해", "그냥 해" 같은 명시적 지시
- 오타 수정 등 명백한 한 줄 수정 (그래도 한 줄 요약은 선행)

## 배경
여러 시도 누적 시 디버깅 추적이 어려워짐. 분석 없이 바로 수정하면 엉뚱한 변경이 쌓여 롤백 비용 증가. 사용자(Wave)의 명시적 피드백 기반 규칙.

## 관련 에이전트
- `bkit-code-analyzer` — 분석 단계에서 능동 활용
- `bkit-frontend-architect` — UI 변경 제안 시 이 룰 따를 것
- `bkit-product-manager` — 플랜 단계에서 근거 제시
- `bkit-report-generator` — 세션 리포트에 "분석 근거" 섹션 포함
