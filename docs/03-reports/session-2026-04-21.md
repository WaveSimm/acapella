## Completed Features

### 1. MIDI A-B Loop Oscillation Fix

**Problem**: When A-B loop was active and currentTime reached point B, the player oscillated between B and B+ε instead of looping back to A. Root cause was Tone.Transport's asynchronous state transitions (`stop` → `seek` → `start`) combined with 100ms polling creating race conditions where stale currentTime values triggered re-entry.

**Solution**: Introduced `abCooldownUntilRef` guard with 600ms debounce window. When loop boundary (A or B) is detected, a cooldown flag prevents setInterval from re-triggering loop logic until Transport stabilizes. Timeout increased from 100ms → 200ms for more responsive UI feedback.

### 3. NWC File Upload + Auto-Conversion Pipeline

**Problem**: Conductors have detailed scores in NoteWorthy Composer (.nwc) format but no way to integrate them with the player. Manual MIDI export + upload is friction-heavy.

**Solution**: Full parser pipeline that accepts .nwc files, extracts notation + lyrics, generates MIDI (for playback) and MusicXML (for visual score), and surfaces both alongside the player.

#### Technical Deep-Dive

**NWC Format Understanding** (V2.75):
- Header: `[NWZ]` + zlib compressed payload
- Decompressed: pipe-delimited commands (`|Clip|Time=0|Endpoint=...`)
- Quoted text fields encoded in UTF-8 OR CP949 (depends on composer's OS/NWC locale setting)
- No formal spec; reverse-engineered via file inspection

**Parser** (`src/lib/nwc/parser.ts` — 450 lines):
- Binary handling: `inflateSync` for zlib decompression
- **Dual-encoding support**: Try UTF-8 strict decode first; on failure, fallback to CP949 (EUC-KR)
- Extract: time/duration info (`Durations` cmd), pitch data (`Note` cmd with `Dur`, `Pitch`), lyrics (`Lyric` cmd)
- Build AST: array of note/rest/time-sig/key-sig objects
- Part extraction: instrument list from `StaffProperties` commands

**MIDI Generator** (`src/lib/nwc/to-midi.ts` — 200 lines):
- Multi-track MIDI (one track per part/staff)
- Tempo from `Tempo` command (BPM)
- Time signature from `TimeSignature` command
- Note-to-pitch conversion: step + octave + accidental (natural/sharp/flat)
- Duration mapping: NWC uses "Divisions" (PPQN analog); convert to MIDI ticks
- Key signature (sharps/flats) embedded in track
- Ties (sustained notes) represented as MIDI note-off + note-on
- Uses `midi-file` npm library for standard-compliant output

**MusicXML Generator** (`src/lib/nwc/to-musicxml.ts` — 280 lines):
- Converts AST to MusicXML 4.0 format (readable by MuseScore, Finale, OSMD)
- Score → Part hierarchy
- Measure grouping by time signature boundaries
- Attributes: clef (treble/alto/bass), time sig, key sig, tempo
- Note placement: pitch (step/alter), duration (type + dots), staff/voice
- Lyrics: syllabic=single/begin/middle/end for proper hyphenation
- Directions: tempo marks, dynamics (optional future)

**Upload Endpoint** (`src/app/api/nwc-upload/route.ts` — 120 lines):
- POST multipart/form-data with .nwc file + songId
- Parse → generate MIDI + MusicXML in memory
- Save UploadedFile records to DB (stores both formats with sourceSite="NWC")
- Create PracticeResource entries for each (MIDI for playback, MusicXML for score)
- Returns parsed metadata (title, composer, tempo, time signature, stave list)

**Score Viewer** (`src/components/practice/score-viewer.tsx` — 180 lines):
- Loads MusicXML via OpenSheetMusicDisplay (OSMD) library
- Features:
  - **System-level viewport**: Measures systems (line groups) by DOM inspection; dynamically adjusts viewport height to show one system at a time
  - **Zoom control**: Default 0.5 (staff size 50%) for mobile readability; adjustable
  - **Part highlighting**: Click part name → filter to that instrument (toggle via `Instrument.Visible`)
  - **Cursor sync**: MIDI currentTime → note position. Formula: `tempo * time / 240` (whole notes per beat, music theory standard)
  - **Responsive scroll**: smooth scroll between systems as MIDI plays
  
**NWC Score Player** (`src/components/practice/nwc-score-player.tsx` — 250 lines):
- Integrates MIDI player + score viewer
- Layout: MIDI controls top, score viewport below
- Syncs currentTime between MIDI engine and score cursor
- Part selector below score (dropdown: "All" / "Soprano" / "Alto" / etc.)
- Transports A-B points from MIDI player to score (highlight measure range)

**Uploader UI** (`src/components/practice/nwc-uploader.tsx`):
- File input (accept=".nwc")
- Shows parsing result (title, composer, parts detected)
- Feedback on success/failure

**Key Technical Insights**:
1. **Encoding challenge**: NWC files created on Mac (UTF-8) vs Windows (CP949) encode text differently. Solution: UTF-8 strict decoder with CP949 fallback catches both. ASCII short-circuit prevents unnecessary iteration.
2. **OSMD system layout**: Score rendered as single large canvas. Identifying system (stave line group) boundaries requires DOM measurement (`g.vf-stave` element inspection). System height varies by content (key changes, clef changes add height).
3. **MIDI-to-score sync**: OSMD doesn't expose a "what note is playing at time X" API. Instead, we calculate using music theory: duration of whole note = 240 ticks (PPQN standard), position = (tempo * currentTime / 60) / (240 / PPQN). Tempo changes mid-score require segment-based calculation.
4. **Part visibility toggle**: OSMD's `Instrument.Visible` flag hides staves; re-render happens automatically. Useful for solo practice (focus one part).

## Technical Decision Summary

### 1. MIDI A-B Loop Synchronization
- **Decision**: 600ms cooldown guard on Tone.Transport state transitions
- **Why**: Tone.js Transport is asynchronous; multiple `start/stop/seek` calls in quick succession cause race conditions. Polling interval (100ms) could fire while Transport is mid-state-change, reading stale `currentTime`.
- **Alternative considered**: Promise-based async loop (await Transport.isRunning). Rejected: Tone.js doesn't expose that API reliably.
- **Outcome**: Stable A-B looping confirmed in production

### 2. Drive API Authentication
- **Decision**: API Key (HTTP Referer restricted) instead of OAuth
- **Why**: Conductor only needs to scan public, self-owned Drive folders. OAuth flow adds complexity; API key sufficient + enforced via HTTP Referer header on Vercel domain.
- **Alternative considered**: Service account (JSON key). Rejected: Unnecessary for read-only public folder access.
- **Trade-off**: API key cannot access private folders, but that's a feature (security boundary).

### 3. NWC Format Choice (vs alternatives)
- **Decision**: Full binary parser for NWC V2.75 instead of relying on external converter
- **Why**: 
  - NWC binary format is straightforward (zlib + pipe-delimited text)
  - No server-side executable (MuseScore CLI, Finale) needed
  - Dual encoding support (UTF-8 + CP949) is essential; built-in decoder doesn't handle that.
  - Full control over MIDI/MusicXML output (can optimize for our use case: playback sync, part isolation)
- **Alternative considered**: System call to MuseScore CLI. Rejected: Adds deployment complexity, licensing questions.

### 4. Score Display: OSMD vs VexFlow
- **Decision**: OSMD (OpenSheetMusicDisplay) for score rendering
- **Why**: 
  - OSMD wraps Vexflow but provides music-theory-aware APIs (Instrument.Visible, tempo queries)
  - Smaller bundle than Finale/MuseScore web SDKs
  - Active maintenance; good MusicXML compliance
- **Alternative considered**: VexFlow directly. Rejected: Lower-level, would require custom MusicXML→VexFlow transpilation.

### 5. Score Viewport Management
- **Decision**: Fixed-height system-unit viewport with dynamic measurement
- **Why**:
  - OSMD renders entire score as one canvas; naive "show current measure" approach jumps around
  - System (5-stave line group) is a natural visual unit for multi-part music
  - DOM measurement (`g.vf-stave` inspection) reliably detects system boundaries
  - Smooth scroll between systems provides better UX than measure-level jumping
- **Alternative considered**: Measure-by-measure carousel. Rejected: Choral music has many measures per system; too granular.

### 6. Idempotency & Drive Sync
- **Decision**: Upsert based on `driveFileId` (unique Drive object ID)
- **Why**: Conductor may re-sync after adding files. Same physical file on Drive should map to same PracticeResource. Using filename is unreliable (rename risk); using filename + songId would create duplicates if song title changes.
- **Trade-off**: If Drive folder is replaced with different folder, old resources aren't auto-deleted (out of scope per PRD). Manual cleanup needed.

## Lessons Learned

### 1. Tone.js Transport is Asynchronous — Plan for Race Conditions
- **What happened**: Initial A-B loop used synchronous `currentTime` polling without guards. When currentTime hit point B, rapid re-entry of loop logic before Transport finished `stop()` caused oscillation.
- **Root cause**: Tone.Transport's `start()` call is async internally; `currentTime` may not be updated immediately.
- **How to apply**: Any real-time audio engine (Web Audio, Tone.js) requires debounce/cooldown guards when making state transitions (play/pause/seek) in tight polling loops. Always assume 100–200ms latency.

### 2. Text Encoding Detection Requires Fallback Chain
- **What happened**: First NWC files parsed successfully in UTF-8, but conductors' files (created on Windows) were CP949-encoded. Parser crashed trying strict UTF-8 decode.
- **Solution**: UTF-8 strict → CP949 fallback. ASCII short-circuit avoids unnecessary decoding for pure English text.
- **How to apply**: When accepting user files with unknown origin (different OS/app versions), always support multiple encodings. Use try/catch decoder pattern: attempt strict format, fallback to legacy format. Document the order (preferred first).

### 3. DOM-Based Layout Measurement is More Reliable Than Trying to Infer from Data
- **What happened**: Initial score viewer tried to calculate system boundaries from measure counts and time signatures. Failed when clef changes or multi-stave instruments added visual height mid-system.
- **Solution**: Measure DOM elements directly (`g.vf-stave` bounding boxes). OSMD renders; then query the actual rendered layout.
- **How to apply**: For complex visual layouts (especially third-party renders like OSMD), don't infer from data model. Always measure rendered DOM after render completes. Use `requestAnimationFrame` to ensure layout is settled.

### 4. Music Theory Conversion Constants Matter
- **What happened**: Initial MIDI-to-score sync was off-by-2x, causing cursor to lag behind played note.
- **Root cause**: Forgot that MIDI uses 240 ticks per quarter note (standard PPQN), not per whole note. Score calculation was: `position = tempo * time / 240`, which should have been `position = (tempo * time) / (240 / PPQN)`.
- **How to apply**: When bridging music notation and playback: 
  - MIDI uses PPQN (pulses per quarter note) = 240 (industry standard)
  - Score uses beat = quarter note, whole note = 4 beats
  - Conversion: `wholeNoteTime = (midiTime / PPQN) / 4`
  - Document these constants prominently in code.

### 5. NWC Binary Format is Simpler Than Expected
- **What happened**: Thought NWC would be complex binary format; actually pipe-delimited text after zlib decompression.
- **Benefit**: Enabled rapid parser implementation (50 lines for core structure). No need for external libraries or reverse-engineering hex dumps.
- **How to apply**: When evaluating proprietary file formats, check if they're publicly documented or reverse-engineered first. NWC V2.75 spec available online; saved 2–3 days of guesswork.

### 6. HTTP Referer Headers Can Cause Silent API Failures
- **What happened**: Drive API calls failed silently with 400 (bad request) in browser → but worked in curl. Root cause: API key restricted to certain HTTP Referer domains; browser was omitting Referer header for cross-site requests.
- **Solution**: Explicitly add `Referer: https://acapella-nine.vercel.app/` header in server-side fetch.
- **How to apply**: When using API key–restricted endpoints, check the GCP Console for HTTP Referer restrictions. If present, ensure your server-side code includes Referer header with the allowed domain.
